## GROUND RULES FOR THIS ENTIRE IMPLEMENTATION

Before writing a single line of code, commit to these principles. They are not suggestions — they are constraints every feature must pass through:

1. **Speed is non-negotiable.** Every new feature must come with a benchmark. If it slows the core scan path by more than 15%, it becomes opt-in via a flag.
2. **Zero mandatory dependencies.** If a new feature requires a dependency, it must be optional and imported lazily inside a try/except. The core `pip install ansede-static` must remain installable with no further installs.
3. **The offline scan path must never touch a network.** Any network feature is opt-in, flagged, and clearly documented as such.
4. **Every new detector needs a synthetic test.** Add a `CVEEntry(...)` to `benchmarks/cve_corpus.py` for every new rule. 100% recall on your corpus is your quality signal.
5. **False positives are a first-class bug.** A finding a developer suppresses three times without acting on is a bug in your tool, not their code. Track suppression rates mentally and design rules conservatively.
6. **Document everything in CHANGELOG.md.** Every PR-equivalent change gets an entry. You are building trust with developers who read changelogs.
7. **Self-scan always passes.** Before every release, run `ansede-static src/ --fail-on high` against the Ansede codebase itself. If it flags your own code, fix the code or the rule — never suppress it.

### 1.1 — Bounded inter-procedural taint tracking (Python)

This is the single highest-value technical improvement you can make. The current taint tracking is intra-procedural — it follows tainted values within a single function. The most dangerous real-world IDOR bugs cross function boundaries.

**The target pattern to detect:**

```python
# Current: Ansede misses this
def get_document_id(request):
    return request.args.get("doc_id")  # tainted source

@app.route("/document")
@login_required
def view_document():
    doc_id = get_document_id(request)   # taint crosses function boundary
    return db.execute("SELECT * FROM docs WHERE id = ?", (doc_id,))  # IDOR: no ownership check
```

**Implementation approach — call-graph taint propagation:**

Add a `CallGraphBuilder` class to `python_analyzer.py`:

```python
class CallGraphBuilder(ast.NodeVisitor):
    """
    Builds a lightweight call graph for a single module.
    Maps function_name -> set of function names it calls.
    Also maps function_name -> whether it returns a tainted value
    (i.e., reads from request.args, request.form, request.json, path params, etc.)
    """
    
    TAINT_SOURCES = {
        "request.args.get", "request.form.get", "request.json",
        "request.get_json", "request.data", "request.values",
        # FastAPI
        "Query", "Path", "Body", "Form", "Header",
    }
    
    def __init__(self):
        self.functions: dict[str, ast.FunctionDef] = {}
        self.taint_returns: set[str] = set()  # functions that return tainted values
        self.call_graph: dict[str, set[str]] = {}
    
    def visit_FunctionDef(self, node):
        self.functions[node.name] = node
        self.call_graph[node.name] = set()
        # Check if this function returns a tainted value
        for child in ast.walk(node):
            if isinstance(child, ast.Return) and child.value:
                if self._expr_is_tainted(child.value):
                    self.taint_returns.add(node.name)
        self.generic_visit(node)
    
    def _expr_is_tainted(self, node) -> bool:
        # Check if an expression reads from a known taint source
        src = ast.unparse(node) if hasattr(ast, 'unparse') else ""
        return any(s in src for s in self.TAINT_SOURCES)
```

Then in your IDOR checker, after collecting local taint sources, extend them with the return values of called functions that are themselves taint sources (up to depth 3 — beyond that the false positive rate climbs):

```python
def _resolve_taint_sources_interprocedural(
    func_node: ast.FunctionDef,
    call_graph: CallGraphBuilder,
    depth: int = 3
) -> set[str]:
    """
    Returns set of variable names in func_node that are tainted,
    including those tainted via helper function calls.
    """
    tainted_vars = set()
    # First pass: direct taint sources
    for node in ast.walk(func_node):
        if isinstance(node, ast.Assign):
            if _is_direct_taint_source(node.value):
                for target in node.targets:
                    if isinstance(target, ast.Name):
                        tainted_vars.add(target.id)
    
    if depth <= 0:
        return tainted_vars
    
    # Second pass: variables assigned from calls to taint-returning functions
    for node in ast.walk(func_node):
        if isinstance(node, ast.Assign) and isinstance(node.value, ast.Call):
            called_name = _get_call_name(node.value)
            if called_name and called_name in call_graph.taint_returns:
                for target in node.targets:
                    if isinstance(target, ast.Name):
                        tainted_vars.add(target.id)
    
    return tainted_vars
```

Important: gate this behind a `--deep` flag initially. Default scan uses the existing intra-procedural logic. `--deep` enables the call graph analysis. This lets you ship it without breaking the speed benchmark, and lets users opt into the slower but more thorough mode.

Add to the CLI:
```

ansede-static src/ --taint-depth 5  # default is 3
```

### 1.2 — Framework detection and scope narrowing

Right now Ansede scans everything the same way. A route-detection rule should only fire on files that actually define routes. Currently if you scan a file of utility functions, you may get noise from heuristics designed for route files.

Add a `FrameworkFingerprint` step that runs once at scan start:

```python
class FrameworkFingerprint:
    """
    Detects which frameworks are present in the codebase by scanning imports.
    Used to activate only relevant rule categories.
    """
    
    SIGNATURES = {
        "flask": ["from flask import", "import flask", "Flask(__name__)"],
        "fastapi": ["from fastapi import", "import fastapi", "FastAPI()"],
        "django": ["from django", "import django", "django.urls"],
        "express": ["require('express')", "require(\"express\")", "from 'express'"],
        "nextjs": ["from 'next/", "next/router", "getServerSideProps"],
        "nestjs": ["@Controller", "@Injectable", "from '@nestjs/"],
    }
    
    @classmethod
    def detect(cls, file_contents: str) -> set[str]:
        detected = set()
        for framework, sigs in cls.SIGNATURES.items():
            if any(sig in file_contents for sig in sigs):
                detected.add(framework)
        return detected
```

Then in each rule function, check the fingerprint before firing:

```python
def _rule_cwe862_missing_auth(ctx: _Ctx):
    # Only run auth decorator checks on Flask/FastAPI files
    if not ctx.frameworks.intersection({"flask", "fastapi", "django"}):
        return  # Skip this rule for non-web files
    # ... rest of rule
```

This will meaningfully reduce false positives on utility files, data processing scripts, and test files.

### 1.3 — Confidence scoring and analysis_kind metadata

Every finding should carry a `confidence` field. Currently the SARIF output includes this, but the text output doesn't surface it clearly. Fix that.

### 1.4 — Django ORM and SQLAlchemy injection patterns

The current SQL injection detection covers f-string and %-format concatenation. It likely misses Django ORM raw query patterns:

## PHASE 3: RULE SYNDICATION — MULTIPLYING COVERAGE WITHOUT MULTIPLYING WORK
### Do this fourth. Estimated effort: 2–3 weeks.

This is the highest-leverage feature for a solo developer. A community rule registry means the world writes your detection rules and you curate them.

### 3.1 — Design the rule format first

Before building any registry infrastructure, define the rule schema precisely. Every community rule must conform to it. YAML is the right format — it's already used by `~/.ansede/community_rules/`.

```yaml
# Example community rule: django-debug-enabled.yaml
schema_version: "1.0"
rule_id: "COMMUNITY-PY-001"
name: "Django DEBUG mode enabled in production"
author: "community"
cwe: "CWE-1188"
severity: "HIGH"
confidence: "HIGH"
language: "python"
frameworks: ["django"]
description: >
  Django's DEBUG=True setting exposes stack traces, environment variables,
  and local variable values to anyone who triggers an error. This must be
  False in production.
pattern:
  type: "ast_assign"
  target: "DEBUG"
  value: "True"
  context: "settings"  # only fire in files matching *settings*.py
test:
  true_positive: |
    DEBUG = True
  true_negative: |
    DEBUG = False
  false_positive_guard: |
    # DEBUG = True  (commented out)
tags: ["django", "configuration", "debug"]
references:
  - "https://docs.djangoproject.com/en/stable/ref/settings/#debug"
```

Key design decisions:
- `schema_version` is mandatory. You will break rules with future schema changes. Version them.
- `test` blocks are mandatory. Any rule without tests is rejected at registry submission time.
- `confidence` is mandatory. Forces rule authors to think about their false positive rate.
- `frameworks` is optional but strongly encouraged — it enables the scope narrowing from Phase 1.2.

### 3.2 — Build the local rule engine first

Before thinking about a remote registry, make the local rule engine excellent. Most community rules will use simple pattern matching. Build a `PatternRuleEngine` that evaluates YAML rules against ASTs:

```python
# src/ansede_static/community_rules/engine.py

class PatternRuleEngine:
    """
    Evaluates YAML-defined community rules against parsed ASTs.
    Supports pattern types: ast_assign, ast_call, regex_line, ast_import
    """
    
    def load_rules(self, rules_dir: Path) -> list[CommunityRule]:
        rules = []
        for path in rules_dir.glob("**/*.yaml"):
            try:
                with open(path) as f:
                    raw = yaml.safe_load(f)
                rule = CommunityRule.from_dict(raw)
                rule.validate()  # Raises if test blocks fail
                rules.append(rule)
            except CommunityRuleError as e:
                # Log warning but don't crash — bad community rules shouldn't break the scan
                logger.warning(f"Skipping invalid rule {path}: {e}")
        return rules
    
    def evaluate(self, rule: CommunityRule, tree: ast.AST, source: str, filename: str) -> list[Finding]:
        if rule.pattern.type == "ast_assign":
            return self._eval_assign(rule, tree)
        elif rule.pattern.type == "ast_call":
            return self._eval_call(rule, tree)
        elif rule.pattern.type == "regex_line":
            return self._eval_regex(rule, source)
        return []
```

This engine must self-validate: when loading a rule, it runs the rule's own `test.true_positive` code through itself and asserts a finding. If the rule's own test fails, the rule is rejected. This prevents broken rules from ever silently failing to detect anything.

### 3.3 — The remote registry (offline-compatible design)

**Architecture decision: pull-at-sync, not pull-at-scan.**

The scan path never touches a network. Rules are always loaded from local disk. Syncing is an explicit, opt-in, separate command:

```bash
ansede-static --sync-rules          # Fetches latest verified rules, saves to ~/.ansede/community_rules/
ansede-static --sync-rules --dry-run # Shows what would be updated without writing
ansede-static --no-community-rules  # Disables all community rules for maximum predictability
```

**Rule registry infrastructure** (minimal viable version):

The registry does not need a custom server. Use GitHub Releases as the backend:

1. Create a new repo: `ansede/community-rules`
2. All community rules live there as YAML files in `rules/<language>/<cwe>/`
3. Each release tag is a signed snapshot of the verified rule set
4. `--sync-rules` fetches the latest release tarball, verifies the SHA256 checksum, extracts to `~/.ansede/community_rules/`

```python
# src/ansede_static/sync.py

REGISTRY_URL = "https://api.github.com/repos/ansede/community-rules/releases/latest"
RULES_CACHE = Path.home() / ".ansede" / "community_rules"

def sync_rules(dry_run: bool = False) -> SyncResult:
    response = requests.get(REGISTRY_URL, timeout=10)
    release = response.json()
    
    tarball_url = release["tarball_url"]
    expected_sha = _get_sha_from_release_notes(release["body"])
    
    tarball = requests.get(tarball_url, timeout=30).content
    actual_sha = hashlib.sha256(tarball).hexdigest()
    
    if actual_sha != expected_sha:
        raise SecurityError(
            f"Rule registry checksum mismatch. Expected {expected_sha}, got {actual_sha}. "
            "Refusing to install potentially tampered rules."
        )
    
    if dry_run:
        return SyncResult(would_add=..., would_update=..., would_remove=...)
    
    # Extract to cache
    _extract_tarball(tarball, RULES_CACHE)
    return SyncResult(added=..., updated=..., removed=...)
```

**Security controls on community rules:**

Rules from the registry run as data, not code. The `PatternRuleEngine` evaluates them through the AST pattern matcher — they never call `eval()`, `exec()`, or `subprocess`. This means a malicious community rule can at worst produce false positives or false negatives, but cannot execute arbitrary code on the user's machine. Document this explicitly in the README as a security property.

Or inline during a scan:
```bash
ansede-static src/ --explain  # After each finding, show a detailed explanation
```

The explanation should cover:
1. What this vulnerability is (2–3 sentences)
2. Why Ansede flagged this specific code (what pattern matched)
3. A concrete before/after code example
4. Real CVE examples of this class of bug
5. Link to MITRE CWE definition

This turns Ansede into a security education tool, not just a linter. Developers who understand why something is flagged are far more likely to fix it and less likely to suppress it.

### 4.3 — `--watch` mode for development

```bash
ansede-static src/ --watch
```

Watches for file changes (using polling or `watchdog` as an optional dep) and re-scans changed files automatically. Shows findings inline in the terminal as you code, without needing the VS Code extension.

```python
# Gate the dependency properly
try:
    from watchdog.observers import Observer
    from watchdog.events import FileSystemEventHandler
    _WATCHDOG_AVAILABLE = True
except ImportError:
    _WATCHDOG_AVAILABLE = False
```

Install with: `pip install ansede-static[watch]`

### 4.4 — Smarter baseline diffing

The current `--baseline` flag compares findings by some identifier. Make sure it's comparing by `(file_path, rule_id, line_number_range)` and NOT just by line number. Line numbers shift every time you add or remove code above a finding, which makes naïve baseline comparisons useless on active codebases.

Add fuzzy matching to the baseline diff:
- A finding is considered "same as baseline" if `rule_id` matches AND the surrounding 3 lines of code are identical (even if the line number changed)
- Only findings with no baseline match are reported as NEW
- Add `--baseline-update` to regenerate the baseline from the current run

```bash
ansede-static src/ --baseline .ansede-baseline.json               # Only shows new findings
ansede-static src/ --baseline .ansede-baseline.json --baseline-update  # Re-saves current state as new baseline
```

### 4.5 — Machine-readable suppression audit

ansede-static src/ --audit-suppressions
```

### Do this in parallel with everything else. Estimated effort: ongoing.

Every technical improvement above is at risk if this project remains a one-person effort. This is the hardest phase because it requires doing things that don't feel like engineering.

### 5.1 — Write a public roadmap

Create `ROADMAP.md` in the repo root. It should have:
- What is shipped (v1.0)
- What is in progress (current phase)
- What is planned next (next phase)
- What is explicitly NOT planned (the things you've decided not to do and why)

The "explicitly not planned" section is the most important. It tells potential contributors where NOT to spend effort and signals that you have a coherent vision, not just a feature wishlist.

### 5.2 — Tag good first issues deliberately

Every rule addition is a good first issue. Tag them clearly:
- `good first issue` — add a detection rule for `[specific pattern]`, here is the existing rule to use as a template, here is the test format, here is how to add a benchmark entry
- `help wanted` — language-specific knowledge needed (e.g. "Add Ruby on Rails CSRF rule — needs Rails expertise")
- `security research` — find real CVEs that match patterns Ansede currently misses

Write a template for `good first issue` rule additions that includes:
1. The CWE to target
2. A link to the MITRE definition
3. A code snippet of the vulnerable pattern
4. A code snippet of the safe pattern
5. A pointer to an existing rule to use as a template
6. The test file to add to

A contributor who can follow a clear template will contribute a correct rule on the first try. That means you spend 10 minutes reviewing instead of 2 hours debugging.

### 5.3 — Write the "How Ansede works" technical document

The biggest barrier to contributing is not knowing how the codebase works. Write `ARCHITECTURE.md`:

```
ARCHITECTURE.md should cover:
- The scan pipeline: file discovery → language detection → parsing → rule execution → output formatting
- How python_analyzer.py works: the _Ctx object, the _rule_NN convention, how rules register
- How js_analyzer.py works: the _Rule pattern vs _check_* contextual functions
- How findings are represented: the Finding dataclass and all its fields

- How the benchmark corpus works

This document is the unlock for contributors. Without it, every new contributor has to reverse-engineer the architecture from code before they can write their first rule.

Enable GitHub Discussions with these categories:

Discussions are searchable. Issues are for bugs and tasks. Discussions are for community. The difference matters for building a health community around the project.

## PHASE 6: OPTIONAL ENTERPRISE LAYER — ONLY AFTER EVERYTHING ABOVE IS DONE
### Do NOT do this until Phases 0–5 are complete. Estimated effort: 8–12 weeks.

Only start this phase when:
- You have at least 500 GitHub stars (signal of real adoption)
- You have at least 3–5 regular contributors
- The false positive rate on real codebases is documented and below 10%
- You have received at least 3 unsolicited requests for enterprise features from real companies

If those conditions aren't met, building a SaaS platform is building infrastructure for users you don't have yet.

### 6.1 — The open-core model

Keep the CLI MIT-licensed and fully featured. The enterprise tier adds:

**Tier: Ansede Pro (individual, ~$15/month)**
- HTML dashboard with finding trends over time (local-only, no cloud)
- SBOM generation (CycloneDX format)
- Priority support

**Tier: Ansede Team (org, ~$50/month per 10 users)**
- Local dashboard server (self-hosted Docker container — never requires sending code to cloud)
- Team-level baseline management (shared baseline across the org)
- Slack/Teams notifications for new findings
- SSO (SAML/OIDC)

**Tier: Ansede Enterprise (custom pricing)**
- Managed cloud dashboard (SARIF ingestion across repositories)
- RBAC for CISOs
- SOC 2 Type II compliance attestation
- SLA

The key principle: **the cloud is never required.** Enterprise customers who cannot send code to the cloud (finance, defence, healthcare) must be able to use every feature except the managed dashboard by running Ansede entirely on-prem.

### 6.2 — JetBrains plugin before SaaS

Before building a cloud dashboard, ship a JetBrains plugin. JetBrains (IntelliJ, PyCharm, WebStorm) is the dominant IDE in the enterprise Java/Python/JS shops you are targeting. The VS Code extension is fine for developers. The JetBrains plugin is what gets you into enterprise standardisation decisions.

The JetBrains plugin does not require a separate backend — it shells out to the locally installed `ansede-static` CLI and renders findings as inspection highlights. The hard work is already done. The plugin is mostly UI plumbing.

### 6.3 — If you build a cloud dashboard, design for data minimisation from day one

The dashboard should:
- Accept SARIF file uploads (findings metadata only — never source code)
- Store findings aggregated by repository, date, severity, and rule_id
- Show trends: is the finding count going up or down over time?
- Support RBAC: engineers see their repo findings, security leads see all repos, CISOs see the organisation-level summary

What it should never do:
- Store source code, line content, or file names (findings contain enough context)
- Use findings data for ML training or product analytics
- Make findings visible to Ansede employees without explicit customer consent

Write this in your Terms of Service from day one. Enterprise security buyers read ToS. Being explicit about what you do NOT collect is a sales advantage against Snyk and Checkmarx.

## WHAT YOU EXPLICITLY SHOULD NOT DO

These are things that sound good but will hurt the project at its current stage:

**Do not add a "100% recall" marketing claim.** You already honestly caveat the synthetic benchmark. That honesty is a competitive advantage with senior security engineers who have been burned by marketing claims. Protect it.

**Do not try to replicate CodeQL's symbolic execution.** You cannot win that fight as a solo developer, and you don't need to. Your moat is IDOR and auth detection that is fast, offline, and zero-config. Double down on that, don't chase CodeQL's feature set.

**Do not add AI/LLM-based scanning as a core feature.** The tool description says "AI-Powered" but the README doesn't back that up. Either implement a specific, well-scoped AI feature (e.g. LLM-generated fix suggestions as an opt-in `--ai-fix` flag that calls a local model) or remove the "AI-Powered" claim. Vague AI marketing erodes trust with the developers you most need to win.

**Do not build the SaaS platform before you have real users asking for it.** You can plan it. Design the schema. Think through the architecture. But don't build infrastructure for a user base you don't have yet.

**Do not sacrifice scan speed for coverage.** The moment `ansede-static src/` takes more than 5 seconds on a typical project, developers start skipping it. Every new feature that touches the hot path needs a benchmark before and after.

## FINAL HONEST ASSESSMENT

Ansede has something genuinely valuable: it finds IDOR, missing auth, and ownership bypass bugs that no other free tool catches out of the box. That is real. That is worth building on.

The risk is not technical. The risk is that a single-contributor project with 3 commits either stalls or tries to do too much too fast and loses the quality that makes it worth using in the first place.

The path to "best SAST tool overall" does not go through feature parity with CodeQL. It goes through being the undisputed best tool at the specific class of web application auth and access control bugs — the bugs that actually appear in CVE databases, that actually cost companies millions, that Bandit and default Semgrep silently miss — and then expanding outward from that moat with the patience and discipline to do each expansion properly.

That is a winnable fight.
